企业认证功能 PRD v7.1

版本：v7.1 开发精简版 更新：2026-04-28 作者：产品部 关联原型：原型_企业认证_PC端_v6.8.html（PC/H5）  原型_企业认证_小程序端_v6.8.html（小程序）

1功能概览

模块说明
企查查企业搜索输入≥2字触发模糊搜索，选中后自动填充：企业名称、统一社会信用代码、法定代表人姓名、注册地址
认证表单用户填写：法人身份证号 + 经办人（姓名/身份证/接收订单消息电话，均必填）。经营类型/公司电话/邮箱选填。协议必勾
草稿保存永久保存，不限时间。下次进入自动恢复；点【重新填写】物理删除草稿
场景一：首次认证企业未在e签宝认证 → 调e签宝接口1（完整参数）→ H5内完成人脸+营业执照+管理员授权
场景二：已认证授权企业已在e签宝认证 → 调e签宝接口1（仅传企业标识）→ H5内仅走管理员授权
统一表单架构商城侧统一展示完整表单（不区分场景一/二）。e签宝H5内部自动识别企业是否已认证并自行适配流程；页面3.5是纯前端过渡动画（setTimeout 1-2秒），不调任何接口
回调处理e签宝认证/授权完成 → notifyUrl回调 → 验签 → 加锁 → 更新状态 → 调接口2同步企业信息
结果页认证+授权通过 / 认证失败 / 认证通过等待授权 / 授权拒绝（最多重申请3次）/ 授权超时
小程序端共用同一套后端接口，e签宝H5通过 <web-view> 内嵌；认证完成通过中间H5回调页跳回小程序；clientType传 MINI_APP

2开发注意事项 & 外部接口链接

⚠️ 关键注意事项

#注意点说明
D1 商城不做场景判断 场景一/二由e签宝H5内部自动识别并切换。商城只需统一走完整表单→调接口1→跳转H5→等回调。页面3.5只是前端过渡动画，不调任何接口
D2 clientType 必须动态传值 PC/H5端传 H5；小程序端传 MINI_APP（后端从Header X-Platform 识别）。传错导致e签宝H5在小程序内渲染异常。禁止生产环境使用 ALL
D3 页面1/2/3.5/4 必须独立路由 不得合并。完整路径：页面1 → 2 → 3.5 → 4 → 5 → e签宝H5 → 7.5 → 结果页
D4 认证失败跳转页面4 点【重新认证】→ 跳页面4（携带已填数据回填），不是草稿恢复页
D5 授权拒绝无撤销功能 页面7B只有【重新申请】，点击弹出修改申请理由弹窗，无取消/撤销
D6 协议勾选必填 未勾选时提交按钮禁用或前端拦截
D7 草稿永久保存 不设时效。90天未更新由后台定时清理，用户不感知
D8 接收订单消息电话无验证码 直接填写，不发短信验证码
D9 回调必须验签 HMAC-SHA256验签，失败返回HTTP 400；验签通过再加分布式锁处理，防重放
D10 场景二双写数据 提交授权申请时同时写 company_auth（AUTHZ_PENDING）和 company_authz_apply 表
D11 小程序回跳：中间H5页 redirectUrl 配置为中间H5页（miniapp-callback.html），内部用 wx.miniProgram.navigateTo 跳回小程序，不可直接配置小程序路径
D12 小程序「税号」= 统一社会信用代码 小程序端标签显示"税号"，后端字段统一为 credit_code，数据完全一致
D13 AccessToken 必须先获取 调e签宝任何接口前先获取AccessToken；建议Redis缓存，过期前5分钟自动刷新
D14 授权重申请最多3次 第4次提交返回400，提示"授权申请已达上限"

3.2 场景一：首次认证（企业未在e签宝认证）

步骤操作说明
1企查查搜索企业，选中后自动填充4字段企业名称/信用代码/法定代表人/注册地址
2页面3.5过渡（1-2秒）→ 完整认证表单（页面4）纯前端，无接口
3填写：法人身份证号 + 经办人（姓名/身份证/接收订单电话）+ 勾选协议接收订单电话无验证码
4提交 → 后端调接口1（传完整 orgAuthConfig + transactorInfo + authorizeConfig）返回 authUrl + authFlowId
5跳转e签宝H5 → 人脸识别 + 营业执照上传 + 管理员授权认证+授权一体化
6e签宝回调 → 验签 → 加锁 → 更新状态 → 调接口2同步企业信息含 organAuthResult + authorizeResult
7根据回调结果跳转结果页通过→AUTHZ_APPROVED；失败→DRAFT可重试；等待授权→AUTHZ_PENDING

接口1传参要点（场景一 vs 场景二对比）：

参数组场景一（首次认证）场景二（已认证授权）
orgAuthConfig → orgName + orgInfo（企业标识）✅ 必传✅ 必传
orgInfo → legalRepName / legalRepIDCardNum（法人信息）✅ 必传❌ 不传
orgAuthPageConfig（认证方式配置）✅ 传（orgDefaultAuthMode=ORG_LEGALREP_INVOLVED）❌ 不传
transactorInfo（经办人信息）✅ 必传（psnAccount=经办人手机号，须与psnMobile一致）❌ 不传
authorizeConfig（授权范围）✅ 必传✅ 必传
clientTypeH5端传 H5；小程序端传 MINI_APP同左
notifyUrl✅ 强烈建议传（不传则无回调，需轮询兜底）同左

3.3 场景二：已认证授权（企业已在e签宝完成实名认证）

场景二运作说明：商城不判断企业是否已认证，统一展示完整表单。e签宝H5识别为已认证企业后，自动跳过认证步骤，仅展示管理员授权页面。
步骤操作说明
1前端流程同场景一（企查查→过渡页→表单页）场景二表单中法人身份证号无需填写
2提交 → 后端调接口1（仅传 orgName + orgInfo 企业标识，不传 transactorInfo/orgAuthPageConfig）e签宝识别已认证企业，返回授权页链接
3跳转e签宝H5 → 仅走管理员授权流程无人脸/营业执照步骤
4e签宝回调 → 更新状态授权通过→AUTHZ_APPROVED；拒绝→AUTHZ_REJECTED（最多重申请3次）
场景二数据写入：提交授权申请时同时写 company_auth（状态AUTHZ_PENDING）和 company_authz_apply 表；法人/经办人字段提交时为 null，e签宝回调后通过接口2同步填充。

3.4 草稿机制

时机/操作说明
自动保存每30秒 + 页面失焦/关闭时
手动保存点击【保存草稿】/ 稍后再说弹窗中的【稍后再说】
草稿有效期永久保存，90天未更新由后台定时清理（用户不感知）
重新填写物理删除草稿 → 回页面1空白表单
继续认证入口认证入口页 / 个人中心 / 签合同时拦截弹窗
认证完成后认证成功回调后自动物理删除草稿

敏感字段（身份证号）只存后端（AES加密），不存本地；非敏感字段同时存本地和后端，以后端为准。

3.5 认证状态机

状态值状态名含义下一步
UNVERIFIED (0)未认证初始状态点【去认证】进入流程
DRAFT (1)草稿有未完成的认证草稿【继续认证】回填 / 【重新填写】清空
VERIFYING (2)认证中已提交，等待e签宝回调等待，可点【刷新状态】
VERIFIED (3)已认证（瞬态）数据库内部态，用户不感知根据授权结果自动跳转
REJECTED (4)认证失败（瞬态）展示失败原因点【重新认证】→ DRAFT → 跳页面4回填
AUTHZ_PENDING (5)授权待处理等待管理员审批等待（无撤销）/ 点【刷新状态】
AUTHZ_APPROVED (6)授权通过认证完成，可签合同【去签合同】
AUTHZ_REJECTED (7)授权拒绝管理员拒绝点【重新申请】→ 弹窗改理由 → AUTHZ_PENDING（最多3次）

4页面说明（PC/H5端）

PC端步骤指示器（4步）：① 填写企业信息 → ② 企业核验 → ③ e签宝实名认证 → ④ 认证完成
📌 页面编号说明：页面1/2/3.5/4 为核心流程页（3.5为纯前端过渡态）；5=e签宝跳转页；7.5=审核等待；8A~8C=场景一结果；7A~7C=场景二授权结果。编号跳跃是历史遗留命名，开发时以路由路径为准。
页面路由核心内容关键交互
页面1：认证入口 /company/auth 展示未认证状态 + 认证权益（账期支付/专属价格/合同签署/子账号管理） 【去认证】→ 页面2；有草稿时展示草稿恢复页
页面2：企业信息填写 /company/auth/form 企业名称搜索框（≥2字触发）+ 统一社会信用代码 + 法定代表人 + 注册地址（后三项自动填充只读） 选中企业后自动填充4字段；【下一步】→ 页面3.5
页面3.5：过渡加载 /company/auth/checking "正在准备认证材料…"加载动画 不调任何接口，setTimeout 1-2秒后自动跳页面4
页面4：完整认证表单 /company/auth/first-time 企业信息（只读回填）+ 法人身份证号（必填）+ 经办人区（姓名/身份证/接收订单电话，均必填）+ 补充信息（经营类型/公司电话/邮箱，选填）+ 协议勾选（必填） 【提交认证】校验通过后禁用+loading；【保存草稿】
页面4弹窗：授权申请（场景二） 弹窗 企业信息（只读）+ 申请人信息（姓名/手机/申请理由，选填）+ 补充信息（选填）+ 协议（必填） 【提交申请】/ 【稍后再说】→ 保存草稿
页面5：跳转e签宝（场景一） /company/auth/redirect 说明认证步骤（人脸/营业执照/管理员授权） 点【前往e签宝认证】→ 打开authUrl；后台轮询回调
页面5：跳转e签宝（场景二） /company/auth/redirect-authz 说明授权步骤 点【前往e签宝授权】→ 打开authUrl
页面7.5：审核中 /company/auth/processing 企业名称 + 提交时间 + 预计5-10分钟 + 动态进度条 每5秒轮询；点【刷新状态】手动查；完成后自动跳结果页
页面8A：认证+授权通过 /company/auth/success ✅ 认证信息 + 已开通权益 【去签合同】
页面8B：认证失败 /company/auth/failed ❌ 失败原因 + 处理建议 【重新认证】→ 状态变DRAFT → 跳页面4（含回填，不经草稿恢复页）
页面8C：等待授权 /company/auth/authz-pending ⏳ 认证企业信息 + 等待管理员授权说明 【刷新状态】；无撤销功能
页面7A：授权通过 /company/auth/authz-success ✅ 企业名称/管理员/审批时间 + 权益说明 【去签合同】
页面7B：授权拒绝 /company/auth/authz-rejected ❌ 拒绝原因 + 处理建议 仅有【重新申请】→ 弹出修改申请理由弹窗 → AUTHZ_PENDING；最多3次
页面7C：授权超时 /company/auth/authz-timeout 等待时长 + 联系管理员提示 【刷新状态】；触发条件：授权申请超24小时未回调

页面栈提示：全流程约6层（74→75→过渡→表单→WebView→结果），在小程序10层限制内。过渡页建议用 wx.redirectTo 而非 wx.navigateTo，避免栈溢出。
📌 小程序端页面编号：页面74=企业信息填写、页面75=法人+经办人表单（对应PC端页面2+4的拆分），其余编号与PC端路由一一对应。

7接口清单

7.1 e签宝侧接口（后端调用）

接口方法+URL必要性调用时机
接口0：获取AccessToken POST /v3/oauth2/access_token ✅ 必需 调e签宝任何接口前。Redis缓存，过期前5分钟刷新
接口1：获取认证&授权页面链接 POST /v3/org-auth/create ✅ 必需 用户提交表单后，获取 authUrl 跳转e签宝H5。场景一传完整参数；场景二仅传企业标识
接口2：查询机构认证信息 GET /v3/org-auth/{authFlowId} ✅ 必需 e签宝回调后，同步企业信息到商城数据库
接口3：查询机构授权信息 GET /v3/org-authorize/{flowId} ⚠️ 兜底 回调丢失时定时轮询补偿

7.2 回调处理流程（按序执行）

步骤操作失败处理
1HMAC-SHA256 验签失败返回 HTTP 400
2获取分布式锁（lock:callback:{authFlowId}）获取失败返回 429
3幂等检查（同一 authFlowId 只处理一次）已处理返回 200
4调接口2查询并同步企业信息失败记录日志，定时补偿（最多3次，间隔5分钟）
5更新 company_auth 状态—
6删除草稿（如有）；返回 200—
回调状态商城状态更新
认证成功 + 授权通过→ AUTHZ_APPROVED
认证成功 + 授权待审批→ AUTHZ_PENDING
认证成功 + 授权拒绝→ AUTHZ_REJECTED
认证失败→ REJECTED（瞬态）→ DRAFT（保留已填数据）

7.3 商城侧接口（前端调用）

所有接口 Header 需携带 Authorization: Bearer {token}，后端从Token解析 user_id，前端无需显式传。

接口路径方法说明
/api/company/searchGET企查查企业模糊搜索（企查查异常时降级为手动输入）
/api/company/auth/submitPOST提交认证，后端调e签宝接口1，返回 authUrl
/api/company/auth/statusGET查询认证状态
/api/company/auth/callbackPOSTe签宝统一回调（仅e签宝调用）
/api/company/auth/infoGET获取认证详情
/api/company/auth/draftGET / POST / DELETE获取/保存/删除草稿（DELETE 为物理删除）
/api/company/auth/draft/checkGET检查是否有草稿，返回 {hasDraft, draftInfo}
/api/company/authz/applyPOST提交授权申请（有待处理申请时不可重复提交）
/api/company/authz/statusGET查询授权申请状态
/api/company/authz/reapplyPOST重新申请授权，body: {applyReason}（最多3次，超限返回400）

8验收要点

8.1 核心流程验收

验收项通过标准
企查查搜索输入≥2字，2秒内返回结果，选中后4字段自动填充
页面3.5纯前端过渡动画，不调任何后端接口
表单必填验证法人身份证/经办人三项/协议均必填；未勾选协议时提交被拦截
接收订单电话无短信验证码流程，直接填写提交
e签宝跳转提交后正确调接口1，获取authUrl并跳转e签宝H5
回调处理回调正确验签→加锁→幂等检查→更新状态→调接口2同步企业信息
认证失败回线展示8B页面 → 点【重新认证】跳页面4并回填，状态变DRAFT，不经草稿恢复页
授权拒绝页面7B只有【重新申请】；点击弹修改理由弹窗；第4次提交返回400
草稿功能永久保存；入口页有继续认证提示；重新填写物理删除草稿；认证成功后自动删除
统一表单架构商城侧不做场景判断；e签宝H5内自动适配场景；场景二接口1不传transactorInfo和orgAuthPageConfig
异常降级e签宝宕机时展示"稍后重试"；企查查宕机时改为手动输入模式
