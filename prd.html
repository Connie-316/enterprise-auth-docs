1功能概览

模块说明
企查查企业搜索输入≥2字触发模糊搜索，选中后自动填充：企业名称、统一社会信用代码、法定代表人姓名、注册地址
认证表单用户填写：法人身份证号 + 经办人（姓名/身份证/接收订单消息电话，均必填）。经营类型/公司电话/邮箱选填。协议必勾
草稿保存永久保存，不限时间。下次进入自动恢复；点【重新填写】物理删除草稿
首次认证（场景一）企业未在e签宝认证 → 调e签宝接口 → H5内完成人脸+营业执照+管理员授权
已认证授权（场景二）企业已在e签宝认证 → 调e签宝接口 → H5内仅走管理员授权
统一表单架构商城侧统一展示完整表单，不区分场景。e签宝H5内部自动识别并适配
回调处理e签宝认证/授权完成 → notifyUrl回调 → 验签 → 加锁 → 更新状态 → 调接口同步企业信息
结果页认证+授权通过 / 认证失败 / 认证通过等待授权 / 授权拒绝（最多重申请3次）/ 授权超时
小程序端共用同一套后端接口，e签宝H5通过 <web-view> 内嵌；clientType传 MINI_APP

2开发注意事项

⚠️ 关键注意点（D1-D14）

#注意点说明
D1 后端判断场景，前端统一表单 前端统一展示表单。用户提交后，后端查 company_auth 表的 auth_status：UNVERIFIED(0)→场景一；其他→场景二。根据场景决定接口1的 businessScene 参数和是否传 transactorInfo
D2 clientType 必须动态传值 PC/H5端传 H5；小程序端传 MINI_APP。传错导致e签宝H5渲染异常
D3 页面必须独立路由 完整路径：页面1 → 2 → 3.5 → 4 → 5 → e签宝H5 → 7.5 → 结果页
D4 认证失败跳转页面4 点【重新认证】→ 跳页面4（携带已填数据回填）
D5 授权拒绝无撤销功能 页面7B只有【重新申请】，点击弹出修改申请理由弹窗
D6 协议勾选必填 未勾选时提交按钮禁用或前端拦截
D7 草稿永久保存 不设时效。90天未更新由后台定时清理
D8 接收订单消息电话无验证码 直接填写，不发短信验证码
D9 回调必须验签 HMAC-SHA256验签，失败返回HTTP 400；验签通过再加分布式锁处理
D10 场景二双写数据 提交授权申请时同时写 company_auth 和 company_authz_apply 表
D11 小程序回跳：中间H5页 redirectUrl 配置为中间H5页，内部用 wx.miniProgram.navigateTo 跳回小程序
D12 小程序「税号」= 统一社会信用代码 小程序端标签显示"税号"，后端字段统一为 credit_code
D13 AccessToken 必须先获取 调e签宝任何接口前先获取AccessToken；建议Redis缓存，过期前5分钟自动刷新
D14 授权重申请最多3次 第4次提交返回400，提示"授权申请已达上限"

3.2 首次认证流程（场景一）

步骤操作说明
1企查查搜索企业，选中后自动填充4字段企业名称/信用代码/法定代表人/注册地址
2页面3.5过渡（1-2秒）→ 完整认证表单（页面4）纯前端，无接口
3填写：法人身份证号 + 经办人（姓名/身份证/接收订单电话）+ 勾选协议接收订单电话无验证码
4提交 → 后端调接口1（传完整参数）返回 authUrl + authFlowId
5跳转e签宝H5 → 人脸识别 + 营业执照上传 + 管理员授权认证+授权一体化
6e签宝回调 → 验签 → 加锁 → 更新状态 → 调接口2同步企业信息含 organAuthResult + authorizeResult
7根据回调结果跳转结果页通过→AUTHZ_APPROVED；失败→页面8B

3.3 已认证授权流程（场景二）

步骤操作说明
1前端流程同场景一（企查查→过渡页→表单页）场景二表单中法人身份证号无需填写
2提交 → 后端调接口1（仅传企业标识）e签宝识别已认证企业，返回授权页链接
3跳转e签宝H5 → 仅走管理员授权流程无人脸/营业执照步骤
4e签宝回调 → 更新状态授权通过→AUTHZ_APPROVED；拒绝→AUTHZ_REJECTED

3.4 草稿机制

时机/操作说明
自动保存每30秒 + 页面失焦/关闭时
手动保存点击【保存草稿】/ 稍后再说弹窗中的【稍后再说】
草稿有效期永久保存，90天未更新由后台定时清理
重新填写物理删除草稿 → 回页面1空白表单
认证入口认证入口页 / 个人中心 / 签合同时拦截弹窗
认证完成后认证成功回调后自动物理删除草稿

敏感字段（身份证号）只存后端（AES加密），不存本地；非敏感字段同时存本地和后端，以后端为准。

3.5 认证状态机

状态值状态名含义下一步
UNVERIFIED (0)未认证初始状态点【去认证】进入流程
DRAFT (1)草稿有未完成的认证草稿【继续认证】回填 / 【重新填写】清空
VERIFYING (2)认证中已提交，等待e签宝回调等待，可点【刷新状态】
REJECTED (4)认证失败展示失败原因点【重新认证】→ 跳页面4回填
AUTHZ_PENDING (5)授权待处理等待管理员审批等待 / 点【刷新状态】
AUTHZ_APPROVED (6)授权通过认证完成，可签合同【去签合同】
AUTHZ_REJECTED (7)授权拒绝管理员拒绝点【重新申请】→ 弹窗改理由（最多3次）

5.2 后端必改项（P0）

改动点说明
clientType 动态识别读取请求 Header 的 X-Platform 字段，小程序端传 MINI_APP，H5端传 H5。传错导致e签宝H5在WebView内渲染异常
redirectUrl 适配小程序端的 redirectUrl 应指向部署在业务域名下的中间H5页（miniapp-callback.html），由该页面调用 wx.miniProgram.navigateTo 跳回小程序
业务域名配置小程序后台配置 webviewUrl 业务域名 openapi.esign.cn，否则 web-view 组件无法加载 e签宝 H5 页面

5.5 e签宝接口调用时序（小程序端）

① 小程序前端 POST /api/company/auth/submit
   Header: { Authorization, X-Platform: miniapp }

② 商城后端 获取AccessToken (HMAC-SHA256鉴权)
   POST /v3/oauth2/access_token
   → 缓存Redis (有效期7200秒)

③ 商城后端 调用e签宝接口1 (获取认证页面链接)
   POST /v3/org-auth/create
   clientType=MINI_APP  ← 关键！
   redirectUrl=https://xxx.com/auth/miniapp-callback.html
   notifyUrl=https://xxx.com/api/company/auth/callback

④ 返回 authUrl → 前端 WebView 加载

⑤ 用户在WebView完成认证 →
   e签宝跳转redirectUrl(中间H5页)

⑥ 中间H5页 wx.miniProgram.navigateTo
   → 跳转小程序结果页

⑦ e签宝异步回调 notifyUrl
   → 后端验签(HMAC-SHA256) → 查询接口2 → 更新数据库

7.1 e签宝侧接口（后端调用）

7.2 回调处理流程

步骤操作失败处理
1HMAC-SHA256 验签失败返回 HTTP 400
2获取分布式锁（lock:callback:{authFlowId}）获取失败返回 429
3幂等检查（同一 authFlowId 只处理一次）已处理返回 200
4调接口2查询并同步企业信息失败记录日志，定时补偿
5更新 company_auth 状态—
6删除草稿（如有）；返回 200—

7.3 场景判断逻辑（后端）

POST /api/company/auth/submit 后：

1. 查询 company_auth 表 WHERE user_id = ?
2. 判断 auth_status：
   ├─ UNVERIFIED(0) → 场景一
   │   ├─ businessScene = FIRST_TIME
   │   └─ 调接口1时传完整参数（含 transactorInfo）
   └─ 其他状态 → 场景二
       ├─ businessScene = ALREADY_AUTH
       └─ 调接口1时不传 transactorInfo（e签宝已有企业信息）

3. 返回 authUrl，前端跳转e签宝H5
